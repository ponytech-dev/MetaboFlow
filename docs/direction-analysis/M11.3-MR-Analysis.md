## 执行摘要

孟德尔随机化（MR）模块是MetaboFlow M11.3的核心功能之一，旨在从观测性代谢组学数据中推断因果关系。本调研的核心结论如下：

**技术可行性**：高。TwoSampleMR R包成熟，OpenGWAS数据库内容丰富，代谢物GWAS数据覆盖率快速提升（近期研究已覆盖1,000+代谢物）。

**实现难点**：中高。OpenGWAS从2024年5月起强制认证，API稳定性存在限速问题；大量代谢物（尤其是非靶向LC-MS鉴定的未知代谢物）缺乏GWAS数据，覆盖率缺口是实质性障碍。

**差异化空间**：窄但存在。MetaboAnalyst 6.0已实现基础MR功能（12种方法），mGWAS-Explorer 2.0已实现18种方法+预计算结果。MetaboFlow的差异化必须来自工作流集成深度，而非方法数量堆砌。

**独立发表价值**：低。功能复制性强，主要作为平台能力，不具备独立论文发表的方法论新颖性。

## 统计方法学详细分析

### 孟德尔随机化的核心假设

MR将遗传变异（SNPs）用作工具变量（IV），通过遗传决定的暴露变化来估计因果效应，规避观察性研究中的混杂和反向因果问题。三个核心假设：

| 假设 | 内容 | 违反后果 |
|------|------|---------|
| **关联性假设** | IV必须与暴露（代谢物）显著关联（F统计量 > 10） | 弱工具变量偏倚，估计值向零收缩 |
| **独立性假设** | IV不受混杂因素影响（遗传随机分配） | 混杂偏倚持续存在 |
| **排他性假设** | IV只通过代谢物影响结局（无水平多效性） | 因果效应估计偏倚 |

### 核心方法比较（证据来源：Genetics in Medicine 2024基准测试研究）

#### IVW（逆方差加权）

**统计基础**: 将多个SNP的比率估计值（β_outcome/β_exposure）按精度加权平均。

**假设**: 平衡多效性（多效效应均值为零）+ InSIDE假设成立。

**优点**:
- 当IV有效时，效能最高
- 可直接使用GWAS汇总统计数据，无需个体水平数据

**缺点**:
- 0%的失效IV下降点（breakdown point）：只要存在有方向性的多效性，估计即产生偏倚
- 在存在无效IV时，p值膨胀最为严重

**适用场景**: 主分析，工具变量经严格筛选后

#### MR-Egger回归

**统计基础**: 在加权线性回归中加入截距项，截距估计多效效应的均值。

**假设**: InSIDE假设（工具变量强度与直接效应独立）；允许定向多效性存在。

**优点**:
- 截距可检验水平多效性（Egger截距检验）
- 在InSIDE假设成立时提供无偏估计

**缺点**:
- 置信区间比IVW更宽，效能显著降低
- 需要工具变量-暴露关联的宽幅变异才能获得精确估计
- InSIDE假设本身难以验证

**典型失效比例**: 即使50%的IV无效，仍可在满足InSIDE假设时提供有效估计。

#### 加权中位数

**统计基础**: 对所有IV的比率估计值按权重排序取中位数，权重与精度成正比。

**假设**: 至少50%的权重来自有效IV（"多数有效"假设）。

**优点**:
- 在多达50%的IV无效时仍保持一致性
- 比MR-Egger效能更高，偏倚比IVW更小

**缺点**:
- 当>20%的IV无效时，I型错误率超过5%标称水平
- 对有效IV的定义依赖于数据本身

#### MR-PRESSO（多效性残差和离群值检验）

**统计基础**: 三步检验——(1)全局检验评估水平多效性存在性；(2)离群值检验识别多效性SNP；(3)去除离群值后的扭曲检验。

**假设**: 允许少数IV存在多效性（离群点模型）。

**优点**:
- 最窄的置信区间（MR-Lasso、MR-Robust、MR-PRESSO类方法）
- 系统性识别并移除多效性离群点
- 提供多效性校正后的效应估计

**缺点**:
- 当多效性SNP数量多且效应量小时，效能下降
- 需要至少3个IV才能运行
- 计算密集（NbDistribution通常设为1000）

### 多方法联合使用策略

标准做法：IVW作为主分析 + MR-Egger + 加权中位数作为敏感性分析 + MR-PRESSO检测离群点。若四种方法方向一致，因果推断可信度最高。

**新兴方法（2024）**:
- **MVMR-cML-SuSiE**: 专门针对代谢物间高度相关性问题，通过识别暴露聚类提升多变量MR的统计效能
- **MR-RIVER**: 针对多血代谢物的效能优化版本

## TwoSampleMR R包功能与API

### 包基本信息

- **维护者**: MRC Integrative Epidemiology Unit (IEU), 布里斯托大学
- **最新版本**: v0.6.20（2025年8月发布）
- **依赖**: ieugwasr（CRAN版本 ≥1.1.0）
- **文档**: https://mrcieu.github.io/TwoSampleMR/

### 核心工作流函数

```r
# 步骤1：列出可用的GWAS数据集
ao <- available_outcomes()

# 步骤2：提取暴露工具变量（代谢物GWAS的显著SNPs）
exposure_dat <- extract_instruments(outcomes = "met-a-XXX")

# 步骤3：提取结局数据（目标表型GWAS中对应SNPs的效应）
outcome_dat <- extract_outcome_data(
  snps = exposure_dat$SNP,
  outcomes = "ieu-a-7"  # 示例：2型糖尿病
)

# 步骤4：数据协调（等位基因对齐、回文SNP处理）
dat <- harmonise_data(exposure_dat, outcome_dat)

# 步骤5：MR分析
res <- mr(dat)

# 步骤6：敏感性分析
res_het <- mr_heterogeneity(dat)      # 异质性检验
res_ple <- mr_pleiotropy_test(dat)    # 多效性检验（Egger截距）
res_single <- mr_singlesnp(dat)      # 单SNP分析

# 步骤7：MR-PRESSO
presso <- run_mr_presso(dat, NbDistribution = 1000)
```

## MR-Base/OpenGWAS API现状

### 数据库规模

- **总数据集**: 超过50,000个GWAS汇总统计数据集
- **代谢物相关GWAS**: 数量持续增长（见下节代谢物覆盖率）
- **访问入口**: https://gwas.mrcieu.ac.uk/ | https://api.opengwas.io/

### 认证系统（2024年5月起强制）

**认证流程**:
1. 在 https://www.opengwas.io 注册账号
2. 生成JWT token（行为类似密码）
3. 在R中设置：`ieugwasr::set_opengwas_jwt("your_token")`
4. 或设置环境变量：`OPENGWAS_JWT`

**账户层级**:
- **Trial层**: 首次注册后默认，访问配额极低
- **Standard层**: 按照网站说明升级，配额更高

### 限速机制（配额制，非传统速率限制）

- 每次查询消耗配额，配额周期性重置（不累积）
- 超出配额返回 `429 Too Many Requests` 错误
- 服务器繁忙时返回 `503` 错误（ieugwasr已实现最多5次自动重试）
- 配额详情在账户页面和响应头中实时可见

### 稳定性评估

**正面因素**:
- 活跃维护（2025年多次CRAN更新）
- 503错误自动重试机制
- 完善的错误传播和日志

**风险因素**:
- 强制认证增加了用户门槛（需要账户注册）
- 配额制在高批量分析时可能成为瓶颈
- API变更频率高（2024-2025年多次接口变动）
- 数据集下载URL仅2小时有效，需要实时获取

**对MetaboFlow的影响**: MetaboFlow需要在用户界面中处理认证流程，且需要设计合理的重试和错误处理逻辑，避免因API限速导致分析中断。

## 代谢物GWAS数据覆盖率现状

### 里程碑研究（按发表时间排序）

| 研究 | 发表年份 | 代谢物数量 | 样本量 | 平台 |
|------|---------|----------|--------|------|
| Kettunen et al. | 2016 | 123 | ~24,000 | NMR |
| Chen et al. | 2023 | 1,091 | 8,299 | LC-MS |
| Nightingale NMR | 2022 | 249 NMR measures | 136,016 | NMR |
| Nature 2024（NMR） | 2024 | 233 | 136,016 | NMR |
| medRxiv 2024 | 2024 | 249 NMR traits | 619,372 | NMR |
| Nature Comms 2025 | 2025 | 249+64比值 | 254,825 | NMR |

### 代谢物GWAS工具变量（IV）可用情况

**mGWAS-Explorer 2.0（2023年发布）**:
- 支持>4,000种代谢物的关联查询
- 但实际可用于MR的有效IV：1,544个IV对应825种代谢物
- 预计算了825种代谢物 × 236种表型的MR结果
- 1,243个显著因果关联已收录

**OpenGWAS数据库中代谢物GWAS**:
- NMR代谢组学：覆盖好（Nightingale平台249个测量值，样本量10万+级别）
- 靶向代谢组学（Biocrates等）：覆盖中等（数百种代谢物）
- 非靶向LC-MS代谢组学：**覆盖极差**

### 关键覆盖率缺口分析

**覆盖好的代谢物**（有公开GWAS，F统计量通常>10）:
- 氨基酸（亮氨酸、异亮氨酸等约20种）
- 脂类代谢物（总胆固醇、HDL/LDL、甘油三酯）
- NMR可测代谢物（乳酸、葡萄糖、丙酮酸等）
- 特定脂质亚类（Nightingale平台）

**覆盖差的代谢物**（缺乏GWAS数据，MR不可行）:
- 非靶向LC-MS发现的大量未知特征（m/z峰）
- 低丰度代谢物（样本量不足，统计效能低）
- 植物/膳食来源代谢物（多酚等）
- 复杂脂质（LC-MS鉴定的数百种脂质亚类，GWAS极少）
- 肠道菌群相关代谢物

**实际覆盖率估算**:
- 靶向NMR代谢组学实验：覆盖率约80-90%（代谢物已知，多有GWAS）
- 靶向LC-MS（Biocrates类）：覆盖率约40-60%
- 非靶向LC-MS：**覆盖率<10%**（绝大多数特征没有GWAS数据）

**结论**: 对于典型的非靶向代谢组学用户，MR模块适用范围极窄。这是功能设计的最大现实障碍。

## MetaboAnalyst 6.0 MR模块具体实现

### 发表信息

- **论文**: "MetaboAnalyst 6.0: towards a unified platform for metabolomics data processing, analysis and interpretation"
- **期刊**: Nucleic Acids Research, 2024（NAR Web Server Issue）
- **DOI**: 10.1093/nar/gkae253
- **作者**: Pang Z, Lu Y, Zhou G 等
- **PMC**: PMC11223798

### MR模块技术架构

**模块名称**: Causal Analysis（因果分析模块）

**底层依赖**:
- TwoSampleMR R包（MRCIEU/TwoSampleMR）
- MRInstruments R包（提供预收录工具变量）
- OpenGWAS API（实时查询GWAS数据）

**工作流程（五步）**:
1. **选择**: 用户从下拉菜单选择暴露（代谢物）和结局（疾病/表型）
2. **工具变量识别**: 系统从mGWAS研究中搜索与代谢物关联的SNPs
3. **SNP筛选与协调**: LD clumping识别独立SNPs；代理SNP替代缺失数据
4. **多效性减少**: 排除影响多种代谢物的SNPs，降低水平多效性
5. **MR分析**: 同时应用12种统计方法

**支持的12种方法**（MetaboAnalyst 6.0）:
- IVW（主分析）
- MR-Egger
- Weighted Median
- 以及另外9种（具体名称论文未全部列出，但MetaboAnalyst网站交互界面有展示）

**输出**:
- 森林图（比较单SNP效应与合并效应）
- 各方法的效应量、置信区间、p值表格
- 异质性检验结果
- 多效性检验结果（Egger截距）
- 交互工具提示解释每种方法的优缺点

**MetaboAnalyst MR的约束**:
- 只能分析MetaboAnalyst数据库中已收录的代谢物（有mGWAS工具变量记录的）
- Web界面，依赖服务器资源，批量分析能力受限
- 用户无法上传自定义GWAS数据
- 不支持多变量MR（MVMR）

## 执行路径规划

### 输入要求

**必需输入**:
1. 暴露：差异代谢物列表（带代谢物名称/HMDB ID/CAS号）
2. 结局：目标表型（疾病名称或OpenGWAS ID，如"ieu-a-7"）
3. OpenGWAS API token（用户自行提供）

**可选输入**:
- 自定义暴露GWAS汇总统计（支持上传自有mGWAS数据）
- 自定义结局GWAS汇总统计（分析非OpenGWAS收录的表型）
- LD参考面板路径（用于clumping）

### 处理流程

```
代谢物列表
    ↓
[名称标准化] → HMDB ID / ChEBI / CAS 映射
    ↓
[工具变量搜索] → OpenGWAS API / 本地mGWAS数据库
    ↓
[可行性筛选] → 判断哪些代谢物有足够IV（F统计量 > 10）
    ↓
[LD Clumping] → 识别独立SNPs（r² < 0.001, 1000G EUR参考）
    ↓
[代理SNP搜索] → 结局GWAS中缺失SNP的替代
    ↓
[等位基因协调] → 处理回文SNPs，对齐效应方向
    ↓
[MR分析] → IVW + MR-Egger + 加权中位数 + MR-PRESSO
    ↓
[敏感性分析] → 异质性检验 + 多效性检验 + 留一法分析
    ↓
[结果可视化] → 森林图 + 散点图 + 漏斗图 + 汇总表
    ↓
输出：代谢物→表型因果关联强度报告
```

### 关键技术决策点

**决策1：工具变量来源**
- 选项A：仅使用OpenGWAS API（覆盖范围依赖数据库）
- 选项B：集成mGWAS-Explorer数据库（825种代谢物，预处理IV）
- 选项C：支持用户上传自定义GWAS汇总统计
- **推荐**: B+C，扩大覆盖范围同时支持高级用户

**决策2：LD Clumping**
- 选项A：调用OpenGWAS API的clumping接口（依赖外部服务）
- 选项B：本地plink2 + 1000G参考面板（稳定但需要额外存储）
- **推荐**: A为默认，B为高级选项

**决策3：API认证处理**
- MetaboFlow需要在用户设置中存储OpenGWAS JWT token
- 需要实现配额监控和友好的限速提示
- 建议预先验证token有效性再启动分析

| MR方法数量 | 12 | 18 | 全部TwoSampleMR | 4+（可扩展） |
| 代谢物覆盖 | 有限（预收录） | >4000（825个有IV） | 依赖用户输入 | 依赖数据源 |

### MetaboFlow的可能差异化方向

**差异化点1：深度工作流集成（最强竞争力）**
MetaboAnalyst的MR是孤立模块，用户需要手动将差异代谢物分析结果带入MR。MetaboFlow可以实现：差异代谢物分析结果 → 自动推送到MR模块 → 批量MR分析 → 汇总报告。这是MetaboAnalyst做不到的。

**差异化点2：本地/私有GWAS数据支持**
MetaboAnalyst和mGWAS-Explorer均不支持用户上传自定义暴露GWAS数据。这对于有自己mGWAS数据的研究团队是实质性需求。

**差异化点3：结果可解释性设计**
针对代谢组学研究者（非遗传学专家）的解读层：
- 自动判断"推荐/谨慎解读/数据不足"
- 清晰展示各方法一致性（颜色编码）
- 内置解读指引（基于方法一致性和F统计量）

**差异化点4：批量分析模式**
支持对所有显著差异代谢物一键批量MR，生成汇总热图（代谢物 × 表型因果关联矩阵）。

## 关键压力测试

### 压测1：MetaboAnalyst已经做了，MetaboFlow再做有何价值？

**分析**:

MetaboAnalyst 6.0 MR模块是Web界面操作，每次分析一对（代谢物×表型），不支持：
- 从同平台差异分析结果直接推送到MR
- 批量分析（如50个差异代谢物 × 5个表型）
- 用户上传自定义mGWAS数据
- 导出原始R脚本以供审稿人核实

MetaboFlow如果做MR，价值不在于"再做一遍"，而在于**打通代谢组学工作流的最后一公里**：从数据处理→统计分析→差异代谢物→因果推断，一个平台内完成闭环。

**结论**: 差异化成立，但仅限于平台集成深度这一维度。单独提供MR功能没有价值。

### 压测2：MR-Base API稳定性

**真实问题**:
- 2024年5月起强制JWT认证，配额制限速
- 配额不累积，高批量分析可能耗尽配额
- 服务器繁忙时503错误（有重试机制）
- API接口变更频率高（2024-2025年多次变动）

**缓解方案**:
- 实现本地缓存：同一SNP/表型查询结果缓存24小时
- 支持本地plink2 clumping，减少API调用
- 支持下载离线数据集（`gwasinfo_files()`接口，2025年新增）
- 用户界面显示配额使用情况

**残留风险**: API政策可能继续变化，MetaboFlow对OpenGWAS存在单点依赖。

### 压测3：代谢物GWAS数据覆盖率（最严重障碍）

**核心数据**:
- mGWAS-Explorer 2.0中，>4000种代谢物查询支持，但实际有有效IV的仅825种
- 非靶向LC-MS发现的代谢物（通常数千个特征）：**绝大多数无GWAS数据**
- 典型非靶向代谢组学实验：差异代谢物50-200个，有MR数据的<20%

**用户体验风险**:
> 用户上传100个差异代谢物 → MR模块告知80个"无可用GWAS工具变量" → 用户体验极差

**缓解方案**:
1. **前置筛选**: 在分析前显示每个代谢物的GWAS数据可用性（绿/黄/红）
2. **扩充数据源**: 集成多个mGWAS数据库（OpenGWAS + mGWAS-Explorer + GWAS Catalog）
3. **透明提示**: 明确说明哪些代谢物为何不能做MR，提供替代分析建议
4. **用户教育**: 区分"没有MR数据"和"MR显示无因果效应"，防止误解

**根本局限无法消除**: NMR代谢组学数据 MR覆盖率好，LC-MS非靶向数据 MR覆盖率差。这是数据本质决定的，不是MetaboFlow设计问题。

### 压测4：零代码用户能否正确解读MR结果？

**方法论复杂性**:
- IVW和MR-Egger方向不一致时如何取舍？
- F统计量<10意味着什么？用户知道吗？
- Egger截距显著说明什么？
- 四种方法都显著是否代表强因果证据？

**典型误解风险**:
1. 将"MR p<0.05"等同于"确认因果关系"
2. 忽视多重检验问题（50个代谢物×5个表型=250次检验）
3. 将弱工具变量的零效应误解为"代谢物无影响"
4. 忽视MR基于欧洲人群GWAS的种族局限性

**缓解设计方案**:
- 结果页顶部显示"综合评级"（绑定多方法一致性+F统计量+样本量）
- 自动多重检验校正（Bonferroni/FDR，显示校正前后p值）
- 内置解读向导：用语言描述"此结果支持/不支持因果关系，因为..."
- 显著性阈值自动标注（建议用户使用FDR<0.05而非p<0.05）

**残留风险不可消除**: MR本身是高级遗传流行病学方法，零代码界面能降低操作门槛，但无法替代遗传学专业知识进行正确解读。最终需要在结果页嵌入足够的方法学提示。

### 压测5：独立发表价值

**结论**: 低。MR功能本身是TwoSampleMR包的包装，技术上无新颖性。发表价值来源于：
- 如果MetaboFlow整体作为平台论文发表，MR是其中一个功能模块
- 如果有具体的代谢组学MR应用研究使用MetaboFlow，可作为工具支撑

MR模块单独发表的可能性：接近零（被NAR 2024 MetaboAnalyst论文抢先发表，mGWAS-Explorer 2.0也已发表）。

## 技术实现方案

### 架构设计

```
MetaboFlow M11.3 MR模块
├── 前端
│   ├── 代谢物-表型配对选择界面
│   ├── GWAS数据可用性预检面板（前置筛选）
│   ├── 参数配置（LD阈值、p值阈值、代理SNP设置）
│   └── 结果可视化（森林图、散点图、漏斗图、汇总表）
├── 后端（R + Python）
│   ├── 代谢物名称标准化（→HMDB/ChEBI ID）
│   ├── OpenGWAS API接口层（含认证、限速处理、重试）
│   ├── 本地mGWAS数据库（mGWAS-Explorer数据导入）
│   ├── LD Clumping（plink2 + 1000G参考）
│   ├── TwoSampleMR调用层（4-12种方法）
│   └── 结果汇总与解读评分
└── 数据层
    ├── OpenGWAS实时API
    ├── 本地mGWAS缓存数据库
    └── 用户上传GWAS数据存储
```

### 核心R包依赖

```r
# 必需
TwoSampleMR >= 0.6.20
ieugwasr >= 1.1.0

# 推荐
MRPRESSO           # MR-PRESSO方法
MendelianRandomization  # 额外方法
```

### 多重检验校正策略

- 单代谢物分析：不校正（方法内比较）
- 批量分析（多代谢物×多表型）：BH-FDR校正
- 结果展示：同时展示原始p值和校正后q值

## 综合评估结论

### 功能必要性

**评分：★★★☆☆（3/5）**

MR模块是代谢组学平台的加分项，但并非核心必需功能。适合有靶向代谢组学数据（NMR/靶向LC-MS）且关注因果推断的研究者。对非靶向LC-MS用户价值有限（覆盖率问题）。

### 技术复杂度

**评分：★★★☆☆（中等）**

TwoSampleMR包成熟，主要复杂度来自：
- OpenGWAS API认证与限速处理（需工程投入）
- 代谢物名称标准化（HMDB映射质量决定覆盖率）
- 结果解读界面设计（需领域专业知识）
- 多重检验校正的合理设置

### 差异化空间

**评分：★★☆☆☆（有限）**

唯一的强差异化点是平台集成深度（从差异代谢物到MR的闭环）。方法层面无法超越mGWAS-Explorer 2.0（18种方法+预计算）。

### 实施建议

1. **优先级**: 中低，建议作为M11.x后期功能迭代，而非核心路线图
2. **MVP范围**: IVW + MR-Egger + 加权中位数 + PRESSO，支持单对和批量分析，集成5-10个高质量代谢物GWAS数据源
3. **必须包含**: 前置GWAS可用性筛选 + 结果解读向导 + 多重检验校正
4. **发表策略**: 作为MetaboFlow整体平台论文的功能之一，不单独发表
5. **数据策略**: 预先导入mGWAS-Explorer 2.0的825×IV数据作为离线默认数据源，减少API依赖
